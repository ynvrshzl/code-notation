# Abstract
An abstract exploration of a theoretical notation language for systems and programming.

Conversation on a notation language for programming and systems.

Studying a little bit of sheet music symbols this morning, we're noticing that sheet music(notation language) emerged as a necessity, not merely abstract rules... e.g. instead of "i will design music as visual" i think the process was more like... "okay, what can we convey as language that other musicians can read and play?"

I think that's like almost a hypothesis... and hypothesis are testable and actionable! whereas... purely abstract ideas are... well... like abstract art... fun to look at... but impossible to read!

So then, if we ever do create this notation thing... lol... we would start with the problems we face as engineers, architects, programmers, or even just plain coding.

- Problem 0: coding by itself is not the issue. it is, arguably, when software emerges into architecture, without proper architecture tools. frustration, burnout, stress, impossible standards. one human is expected to maintain giant infrastructure. physically, that would be inhumane, in coding? it's expected as a baseline prerequisite.
- Problem 1: invisible data flow - data flow is abstract, visually-natured, and yet, in code, no way to track it without jumping between contexts and files. that level of cognitive distortion is unsustainable. It might explain the majority of Problem 0.
- Problem 2: abstraction requires record-keeping (tracking which abstraction layers are responsible for data flow)

our language would have to be;

- cognitiively conceptually useful
- not require any translation layer
- solve the problems we mentioned. especially the invisible data flow 
- no formal symbols, just using plain-text. (maybe)
- not executuable (we're not re-creating programming lol.)
- language that actually solve problems, in the same way, other fields used  notation languages to assist human cognitive gaps. 

the mechanics of language, most-likely explains why it is so useful for us human beings:
- language = orientation in audio, cognitive, and visual space
- language, is our most reliable source for encoding *thought* between human beings. ideas travel in time and space, language is the medium for them.

sample? this is purely exploratory, not meaningful at all! just visual gibberish to get an idea of what notation for systems would look like, if it were written. without any formal symbols, just using plain-text. 

note, that all of this is taken from my own personal observations and experiences, of what is actually useful, while working with deeply complex, interlayered systems. they resemble computer-cpu architecture, and we should approach them as such! 

Some conceptual schematics...
(Aa) Alpha: (this is the beginning of the system. perhaps it is the problem to solve, or the goal in mind. the key here is that all sustainable systems must have clearly defined conceptual boundaries. the alpha is the beginning of the system. the origin of where things emerge from and return to. )

Thesis: (this is the formal structure of all computer programs. it's like the scientific method hypothesis model. starting a system with a question, rather than a problem to solve.) Thesis is denoted by (t...)

e.g.

(t...) how can we create a cryptographic password generator, that can decrypt the result, securely?


...that's inherently an architectural thesis/theorem/hypothesis... multilayered, but still conceptually useful! it could actually guide a process.

some more models!

(hyp) Hypothetical/Hypothesis: would be a proposition for a model, to implement in code. This helps us avoid getting lost in the details of the code, and stay in our native, cognitive-map mode. This is probably where most of the notation language would exist. To compress ideas into implementable programming models.

i think our notation language might actually also contain a hypothetical model for programming systems... and this way, it can also fit in human working memory.. so it's portable and conceptual!

approaching a system like this: "(t...) what can i do to make (Aa) happen?" instead of "wait, why is my syntax highlighting the wrong keyword?" takes the cognitive human lens AWAY from the medium (the ide, the programming language, the technology) and focuses the cognitiion onto the system. just like a civil engineer and an architect would do so... according to what we know of those fields.
